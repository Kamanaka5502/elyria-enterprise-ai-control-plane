# Design Language

## Purpose

This document defines the public-facing visual and language system for the Elyria Enterprise AI Control Plane.

The goal is to make the repository read as a polished enterprise architecture asset: serious, clean, board-readable, and suitable for consulting, AI governance, Azure strategy, responsible AI, and pilot-readiness conversations.

---

## Visual Positioning

```text
Enterprise-grade
Azure-aligned
Governance-forward
Pilot-ready
Public-safe
Executive-readable
Delivery-usable
```

---

## Palette

| Role | Color | Use |
|---|---|---|
| Deep graphite | `#1f4f5a` | Primary structure, architecture, control-plane framing |
| Slate teal | `#2f6f73` | Responsible AI, agentic governance, security language |
| Muted steel | `#5f8fa3` | Azure alignment, observability, platform references |
| Warm gold | `#c9a66b` | Pilot-ready, executive emphasis, readiness signals |
| Ivory | `#f7f2e8` | Background concept and clean report tone |

---

## Badge System

Use badges to signal enterprise credibility quickly:

```text
License
Enterprise AI
Responsible AI
Azure Alignment
Pilot Ready
Agents
RAG
Data Scope
Decision Model
```

Badges should reinforce architecture value, not decorate without purpose.

---

## Voice

Use language that is:

- precise
- executive-readable
- architecture-oriented
- outcome-driven
- public-safe
- serious enough for enterprise review

Avoid language that sounds speculative, mystical, exaggerated, or implementation-claiming beyond what the repo actually provides.

---

## Core Phrases

Approved public-facing phrases:

```text
Govern AI movement before production consequence.
Turn responsible AI from policy into production movement control.
Public-safe reference architecture.
Pilot-ready sandbox deployment model.
ADMIT / HOLD / REFUSE / REVALIDATE.
Executive-readable. Delivery-usable. Reusable across accounts.
```

---

## Public Boundary Rule

The visual system can be polished and high-trust, but it must not imply that this public repository exposes private Elyria Systems machinery, protected validators, customer implementations, or commercial proof-corridor internals.

The repository should show architecture judgment and delivery readiness while preserving the protected system boundary.
